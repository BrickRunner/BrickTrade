# Проблемы режима торговли акциями

## Дата: 2026-03-30

---

## КРИТИЧЕСКИЕ ПРОБЛЕМЫ

### 1. DivergenceStrategy использует current_rsi вместо RSI в точках свингов
**Файл:** `stocks/strategies/divergence.py:78-80`
Дивергенция сравнивает цену в swing-точках, но RSI берёт ТЕКУЩИЙ (`current_rsi`),
а не RSI в моменты свингов. Это делает стратегию неточной — она может генерировать
ложные сигналы дивергенции. Нужно вычислять RSI-серию по свечам и брать значения
RSI на индексах swing_highs_idx / swing_lows_idx.

### 2. Нет бэктестинга для стратегий акций
Модуль `stocks/` не имеет никакого backtesting-фреймворка. Стратегии невозможно
протестировать на исторических данных. Нельзя оценить ожидаемую доходность, drawdown,
Sharpe ratio. Запуск в auto-режиме без бэктеста — это слепая торговля.

### 3. Нет интеграционных тестов с BCS API
Весь модуль `stocks/exchange/` (bcs_rest.py, bcs_ws.py, bcs_auth.py) не имеет тестов.
Ордера отправляются на реальную биржу без предварительной проверки формата запросов.

## СЕРЬЁЗНЫЕ ПРОБЛЕМЫ

### 4. MeanReversion — не является полноценной парной торговлей
**Файл:** `stocks/strategies/mean_reversion.py`
Стратегия отслеживает ratio двух тикеров, но генерирует интент только на ОДНУ ногу
(только тикер A). Вторая нога пары (B) не открывается. Это значит что при zscore>2
система продаёт SBER, но не покупает SBERP — получается направленная сделка,
а не хеджированная парная торговля. Нужен механизм двухногих ордеров.

### 5. Отсутствует проверка позиций на стороне брокера
**Файл:** `stocks/system/state.py`
Позиции хранятся только в памяти + JSON файл. Нет сверки с реальными позициями
через BCS API (`get_positions()`). При рестарте бота позиции могут разойтись с реальностью.
factory.py вызывает `state.load_positions()`, но не сверяет с BCS.

### 6. _wait_fill может зациклиться при частичном исполнении
**Файл:** `stocks/system/execution.py:207-247`
Метод ждёт статус "2" (filled) или "4"/"8" (cancelled/rejected). Но BCS может
вернуть статус частичного исполнения, который не обрабатывается.
Также нет обработки ситуации когда ордер "повис" — нет cancel при timeout.

### 7. Trailing stop мутирует frozen dataclass
**Файл:** `stocks/system/engine.py:322-335`
В `_process_open_positions()` напрямую присваивается `pos.peak_price = price`
и `pos.stop_loss_price = round(new_sl, 4)`. StockPosition — НЕ frozen, но это
прямая мутация объекта без вызова `_persist_positions()`. Если бот упадёт после
обновления peak_price но до записи — состояние будет потеряно.

### 8. Нет ограничения на количество concurrent подтверждений
**Файл:** `stocks/system/confirmation.py`
В semi_auto режиме каждый сигнал создаёт `asyncio.create_task(_confirm_and_execute)`.
При 25 тикерах × 6 стратегий = потенциально 150 одновременных запросов подтверждения
к пользователю. Нет ни лимита, ни очереди.

## СРЕДНИЕ ПРОБЛЕМЫ

### 9. Индикаторы зависят от market_intelligence модуля
**Файл:** `stocks/system/price_buffer.py:8-21`
CandleBufferManager импортирует 12 индикаторных функций из `market_intelligence/indicators.py`.
Это создаёт жёсткую зависимость между модулями. Если market_intelligence сломается —
акции тоже перестанут работать. Нужна своя копия или абстракция.

### 10. run_forever спит неправильно при закрытом рынке
**Файл:** `stocks/system/engine.py:49`
`seconds_until_close()` возвращает 0 когда рынок закрыт. Код делает
`min(60, max(1, 0))` = 1 секунду. Получается busy-loop каждую секунду всю ночь.
Нужно использовать `next_session_open()` и вычислять реальное время до открытия.

### 11. Все стратегии генерируют quantity_lots=1 по умолчанию
Ни одна стратегия не рассчитывает размер позиции на основе волатильности (ATR)
или risk-per-trade. Всегда 1 лот. Для VTBR (лот=10000 акций) и SBER (лот=10 акций)
это кардинально разные суммы.

### 12. BCS WebSocket не реконнектится при обрыве
**Файл:** `stocks/exchange/bcs_ws.py`
Нет автоматического переподключения при потере соединения.
Если WS упадёт — данные перестанут обновляться, но engine продолжит работать
со стухшими котировками (пока не сработает stale data check через 120с).

### 13. Нет логирования P&L в базу данных
Вся статистика (trades, P&L, drawdown) хранится только в памяти.
При рестарте всё теряется. Нет возможности анализировать историю сделок,
строить equity curve или оптимизировать параметры.

### 14. confirmation.on_modify пересоздаёт intent вместо replace()
**Файл:** `stocks/system/confirmation.py:161-174`
Ручная конструкция нового StockTradeIntent — если добавятся новые поля,
они будут потеряны. Нужно использовать `dataclasses.replace()`.

### 15. Нет проверки сессии при генерации сигналов
Все стратегии работают одинаково в утреннюю, основную и вечернюю сессии.
Но утром (06:50-09:50) торгуются не все бумаги и ликвидность значительно ниже.
VolumeSpikeStrategy может давать ложные сигналы при низкой ликвидности.

### 16. max_total_exposure_pct и max_per_position_pct = 1.0 (100%)
**Файл:** `stocks/system/config.py:37-38`
Дефолтные значения позволяют вложить 100% капитала в одну позицию.
Это слишком агрессивно для стратегий с 3-4% стоп-лоссом.

## КАК РАЗВИВАТЬ РЕЖИМ ДАЛЬШЕ

### Приоритет 1 — Фундамент (1-2 недели)

**1.1. Backtesting фреймворк**
Создать `stocks/system/backtest/` с возможностью:
- Загружать исторические свечи MOEX через BCS API
- Прогонять стратегии на исторических данных
- Считать метрики: Sharpe, Sortino, max DD, win rate, profit factor
- Отключать стратегии с отрицательным expectancy

**1.2. База данных для сделок**
SQLite таблица `stock_trades` с полями:
position_id, strategy, ticker, side, entry_price, exit_price,
lot_size, quantity, pnl, commission, opened_at, closed_at, reason.
Команда /stock_history в Telegram для просмотра.

**1.3. Сверка позиций с BCS**
При старте и каждые N минут — вызов `bcs_client.get_positions()`,
сравнение с `state._positions`. Alert если расхождение.

### Приоритет 2 — Качество стратегий (2-3 недели)

**2.1. Исправить DivergenceStrategy**
Вычислять RSI-серию через candles, использовать RSI[swing_idx]
вместо current_rsi. Добавить минимальное расстояние между свингами.

**2.2. Полноценная парная торговля для MeanReversion**
Генерировать два интента (sell A + buy B) одновременно.
Добавить в модель `StockPairIntent` или группировку интентов.
Исполнять обе ноги атомарно.

**2.3. Position sizing на основе ATR**
Формула: `qty = (equity * risk_pct) / (ATR * atr_mult * lot_size)`.
Стратегии должны возвращать рекомендуемый размер, а не всегда 1.

**2.4. Адаптация к сессиям MOEX**
Утренняя сессия — только мониторинг или reduced sizing.
Вечерняя — без VolumeSpikeStrategy (объёмы низкие).
Передавать session_type в snapshot, стратегии учитывают его.

### Приоритет 3 — Надёжность (2-3 недели)

**3.1. WS auto-reconnect**
Обернуть bcs_ws.py в reconnect-loop с exponential backoff.
Логировать disconnects и reconnects. Alert в Telegram.

**3.2. Обработка partial fills**
В _wait_fill: обрабатывать статус "1" (partial).
При таймауте — отменять ордер через cancel_order().
Позицию создавать на фактически исполненный объём.

**3.3. Persist trailing stop обновления**
После каждого цикла _process_open_positions вызывать
state._persist_positions(). Или переделать на event-driven.

**3.4. Лимит concurrent confirmations**
Семафор на 3-5 одновременных подтверждений.
Остальные в очередь с приоритетом по confidence.

### Приоритет 4 — Масштабирование (4+ недели)

**4.1. Лимитные ордера**
Сейчас все ордера market. Добавить limit ордера
с ценой = last + spread%. Меньше slippage.

**4.2. Multi-timeframe анализ**
Сейчас только M5 свечи. Добавить M15, H1 для подтверждения.
Тренд на H1 + вход на M5 = лучшее качество сигналов.

**4.3. Интеграция с market_intelligence**
Использовать regime detector для отключения стратегий
в неподходящих режимах рынка (trending vs ranging).

**4.4. Dashboard с equity curve**
Веб-страница или Telegram inline с графиком equity,
PnL по стратегиям, win rate, drawdown timeline.
