# Отчёт: 14 улучшений арбитражного бота

## Обзор

Проведена комплексная доработка арбитражного движка OKX/HTX. Все 14 проблем исправлены.

---

## Проблемы прибыльности (1-8)

### 1. Учёт funding rate в PnL
**Было:** Funding rate не учитывался — реальный PnL мог быть значительно ниже расчётного.
**Стало:** `_update_funding_rates()` загружает ставки с обеих бирж каждые ~120 циклов. `_estimate_funding_cost()` рассчитывает стоимость финансирования за время удержания. При закрытии сделки funding cost вычитается из PnL.

### 2. Восстановление позиций после перезапуска
**Было:** При рестарте бота открытые позиции терялись — orphaned позиции на биржах.
**Стало:** `_recover_positions()` при инициализации проверяет БД на открытые сделки, затем сверяет с реальными позициями на OKX и HTX. Три сценария: обе есть → восстановление, одна есть → уведомление об orphaned, ни одной → закрытие записи в БД.

### 3. Трекинг реального slippage и комиссий
**Было:** PnL считался по теоретическим ценам, slippage игнорировался.
**Стало:** `ActiveTrade` хранит `actual_long_fill_price` и `actual_short_fill_price`. После входа в сделку запрашиваются фактические цены исполнения (OKX `avgPx`, HTX `cost_open`). Slippage считается и сохраняется в БД.

### 4. Динамический размер позиции
**Было:** Фиксированный `max_position_pct` для всех сделок.
**Стало:** `_calculate_dynamic_position_pct()` учитывает:
- Серию проигрышей: 2 подряд → 75%, 3+ → 50% от базового
- Силу сигнала: spread/entry_threshold ≥ 2.0 → до 125%, ≥ 1.5 → до 110%

### 5. Защита от ликвидации
**Было:** Никакой проверки — при высоком плече возможна ликвидация.
**Стало:** `_check_liquidation_risk()` перед каждым входом проверяет:
- Расстояние до ликвидации ≥ 4% (иначе отказ)
- Свободная маржа ≥ 30% после открытия (иначе отказ)

### 6. Абстракция бирж (multi-exchange)
**Было:** Жёсткая привязка к OKX и HTX в коде.
**Стало:** `get_supported_exchanges()` и `get_exchange_client(name)` — базовая абстракция для добавления новых бирж. Клиенты доступны по имени.

### 7. Spot-futures basis мониторинг
**Было:** Не отслеживался.
**Стало:** `_update_basis_spreads()` каждые ~60 циклов загружает spot тикеры с обеих бирж и считает basis = (futures - spot) / spot * 100%. Доступно через кнопку "Basis" в Telegram. Показывает contango/backwardation для каждой пары.

### 8. Auto-blacklist убыточных пар
**Было:** Пара могла бесконечно генерировать убытки.
**Стало:** `_periodic_blacklist_check()` каждые ~120 циклов проверяет:
- Win rate < 30% при 5+ сделках → блокировка на 24ч
- 3+ убыточных сделки подряд → блокировка на 24ч
- Заблокированные пары исключаются из `_filter_pairs()` и `_scan_for_entry()`
- Данные в SQLite таблице `pair_blacklist` с auto-expire

## Проблемы удобства (9-14)

### 9. Кнопка экстренного закрытия
**Было:** Невозможно быстро закрыть все позиции из Telegram.
**Стало:** Кнопка "Закрыть всё" в меню → подтверждение → `emergency_close_all()`. Закрывает active_trade + сканирует все позиции на обеих биржах и закрывает по рынку.

### 10. Настройки через Telegram (без рестарта)
**Было:** Изменение entry/exit threshold требовало редактирования .env и перезапуска.
**Стало:** `runtime_settings` dict в движке + кнопка "Настройки" в Telegram. Можно менять кнопками +/-:
- Entry threshold (±0.05%)
- Exit threshold (±0.01%)
- Position size (±5%)
- Leverage (±5x)
- Stop-loss (±0.5%)
Изменения применяются мгновенно, с валидацией лимитов.

### 11. Per-pair статистика
**Было:** Только общий PnL без разбивки по парам.
**Стало:** Кнопка "Per-pair" показывает:
- Из БД: сделки, win rate, PnL, fees, funding по каждой паре
- Live: adaptive thresholds (entry/exit), mean/std спреда, blacklist статус
- `get_pair_adaptive_info()` метод для программного доступа

### 12. Сохранение истории сделок в БД
**Было:** История только in-memory, терялась при рестарте.
**Стало:** Новый модуль `arbitrage/core/trade_history.py`:
- Таблица `trades`: все поля сделки (symbol, exchanges, spreads, PnL, fees, funding, slippage, duration, reason)
- Таблица `balance_snapshots`: периодические снимки балансов
- Таблица `pair_blacklist`: авто-блокировка пар
- Функции: `save_trade_open()`, `save_trade_close()`, `get_pair_stats()`, `get_overall_stats()`, `get_recent_trades()`, `export_trades_csv()`

### 13. Live dashboard с adaptive thresholds
**Было:** Показывал только глобальные пороги.
**Стало:** В "Возможности" для каждой пары показывается её adaptive entry threshold. В "Статистике" — полные runtime settings и funding estimate для активной сделки. В "Per-pair" — mean/std/samples для adaptive порогов.

### 14. Уведомления о состоянии баланса
**Было:** Баланс показывался только по запросу.
**Стало:** `_check_balance_alerts()` автоматически:
- Периодический отчёт каждый час (баланс, изменение от начала, PnL)
- Критическое уведомление при падении ≥ 10%
- Снэпшоты в БД каждые 5 минут для истории

## Безопасность

- Все глобальные пороги остаются как floor (adaptive не может быть ниже)
- Emergency close требует подтверждения
- Runtime settings имеют жёсткие лимиты (leverage 1-75x, position 5-80%, и т.д.)
- Liquidation check перед каждым входом
- Circuit breaker для HTX сохранён
- HTX-first sequential order placement сохранён
