## 2. ПОЧЕМУ НЕТ СДЕЛОК?

### 🔴 ПРОБЛЕМА: Слишком строгие фильтры

**Статус позиций**: `data/short_positions.json` → `[]` (пустой)

**Причина**: Сигналы **КРАЙНЕ РЕДКО** проходят все фильтры.

### Жесткие условия для входа

#### Фильтр 1: Score ≥ 5/7 (из EXEC_CONFIG)
Нужно набрать **минимум 5 сигналов из 7**:

| Сигнал | Условие | Сложность |
|--------|---------|-----------|
| 1. RSI > 80 | RSI(14) > 80 | 🟡 Средняя (бывает редко) |
| 2. Price Spike | +5% за 10 свечей | 🔴 Высокая (экстремально) |
| 3. Volume Spike | Объем > 2× средний | 🟡 Средняя |
| 4. Funding > 0 | Funding rate > 0 | 🟢 Легкая (почти всегда) |
| 5. OI Growth | OI растет за 5 периодов | 🟡 Средняя |
| 6. EMA Deviation | Цена > EMA50 на 3% | 🟡 Средняя |
| 7. Divergence | Price high + RSI падает | 🔴 Высокая |

**Реальность**: Получить 5+ сигналов одновременно = **очень редко** (1-2 раза в неделю на всех 80+ парах).

#### Фильтр 2: Обязателен Price Spike (код: 188-189)

```python
if not signal_price:
    is_overheated = False  # ← ЖЕСТКИЙ ФИЛЬТР
```

**Требование**: Цена **ОБЯЗАТЕЛЬНО** должна вырасти на +5% за 10 свечей.

**Проблема**: При таймфрейме 5m это означает +5% за 50 минут — **экстремально редко**.

#### Фильтр 3: Trend Filter (код: 183-185)

```python
if ema_slow_series and closes[-1] > ema_slow_series[-1]:
    if score < cfg["trend_filter_min_score"]:  # 6
        is_overheated = False
```

**Логика**: Если цена **выше** EMA200 (восходящий тренд), нужен score ≥ **6/7** вместо 5/7.

**Проблема**: Это еще больше сужает возможности входа в лонг-трендах.

## 3. ОЦЕНКА СИСТЕМЫ

### ✅ СИЛЬНЫЕ СТОРОНЫ

#### 1. Профессиональная архитектура
- Чистый код, хорошая структура
- Разделение логики (детектор, исполнение, UI)
- Персистентность позиций (`short_positions.json`)
- Полноценный risk-менеджмент

#### 2. Качественная стратегия "Overheat Detector"
**7 комплексных сигналов**:
- RSI: Классический индикатор перекупленности
- Price Spike: Экстремальный рост цены
- Volume Spike: Аномальный объем (возможная кульминация)
- Funding Rate: Sentiment long-трейдеров (платят за позиции)
- Open Interest: Накопление позиций
- EMA Deviation: Отклонение от тренда
- Divergence: Медвежье расхождение (цена растет, RSI падает)

**Логика**: Ищет "перегретые" монеты, которые выросли слишком быстро → вероятен откат → SHORT.

#### 3. Солидный риск-менеджмент
- **Размер позиции**: 5.5 USDT (минималистично)
- **Leverage**: 5× (умеренное плечо)
- **SL/TP**: 2% / 3% (адекватное соотношение риск/профит)
- **Лимит позиций**: Максимум 3 SHORT одновременно
- **Auto-execute**: Автоматический вход при наличии сигнала

#### 4. Bybit API интеграция
- Корректное использование API (REST + WebSocket готовы)
- Правильный расчет lot size / tick size
- Set leverage через API
- Trading stop для SL/TP
- Reduce-only для закрытия

#### 5. Telegram UI
- Удобное меню с кнопками
- Настройка параметров "на лету"
- Мониторинг позиций в реальном времени
- Уведомления о входах/выходах

### ⚠️ СЛАБЫЕ СТОРОНЫ

#### 1. 🔴 КРИТИЧНО: Слишком строгие фильтры

**Проблема**: Score ≥ 5/7 + обязательный Price Spike +5% = сигналы **крайне редки**.

**Последствия**:
- Бот может **неделями не торговать**
- Упускаются умеренные перегревы (score 3-4), которые тоже прибыльны
- На 80+ парах генерируется 0-2 сигнала в неделю

**Решение**: Снизить пороги (см. раздел "Рекомендации").

#### 2. Фиксированные параметры детектора

```python
DEFAULT_CONFIG = {
    "rsi_threshold": 80,           # Очень высоко
    "price_change_threshold": 5.0,  # Экстремально высоко
    "score_threshold": 4,          # Но код использует min_score=5!
    "trend_filter_min_score": 6,   # Еще выше
}
```

**Проблема**: Пороги не адаптируются под волатильность рынка.

**Пример**: В flat-рынке RSI > 80 = очень редко, в волатильном — часто, но быстро откатывается.

#### 3. Отсутствие бэктестинга

**Нет данных**:
- Win rate неизвестен
- Средний P&L на сделку?
- Оптимальные параметры (RSI threshold, price_change)?
- Sharpe ratio, Max DD?

**Риск**: Стратегия может быть убыточной при реальной торговле.

#### 4. Малый размер позиции (5.5 USDT)

**При leverage 5×**:
- Экспозиция = 5.5 × 5 = **27.5 USDT**
- SL 2% = убыток ~**0.55 USDT**
- TP 3% = прибыль ~**0.83 USDT**

**Комиссии Bybit**:
- Maker: 0.02%, Taker: 0.055%
- На вход+выход (market orders) = ~0.11% × 27.5 = **0.03 USDT**

**Чистая прибыль при TP**: 0.83 - 0.03 = **0.80 USDT** (145% к риску)

**Вывод**: Размер адекватен для тестирования, но для прибыли нужно >100 сделок в год.

#### 5. Нет защиты от черных лебедей

**Отсутствуют**:
- Max Daily DD limit (нет kill-switch)
- Max loss streak (подряд 5 убытков → пауза)
- Correlation check (все 3 SHORT в одной секте = катастрофа при pump)

**Риск**: 3 SHORT на мемкоинах во время meme season = -30% за день.

#### 6. Автосканирование каждые 5 минут

```python
_auto_interval: int = 300  # 5 минут
```

**Таймфрейм**: 5m свечи + сканирование каждые 5 минут.

**Проблема**: Для перегревов 5m — слишком частый TF. Лучше 15m-1H для структурных сигналов.

#### 8. Correlation filter

**Проблема**: Открыть 3 SHORT на DOGE, SHIB, PEPE одновременно = катастрофа при meme pump.

**Решение**:
```python
def check_sector_exposure(symbol: str, open_positions: List) -> bool:
    """Limit shorts in same sector."""
    MEME_COINS = {"DOGE", "SHIB", "PEPE", "FLOKI", "BONK"}
    if symbol in MEME_COINS:
        meme_count = sum(1 for p in open_positions
                         if p.symbol in MEME_COINS and p.status == "open")
        if meme_count >= 2:
            return False  # Max 2 meme shorts
    return True
```

**Текущее**: Фиксированный TP 3%.

**Улучшение**: Trailing stop для фиксации прибыли при сильных движениях.

#### 11. Multi-timeframe confirmation

    return True  # Открыть SHORT

**Преимущество**: Фильтрует ложные сигналы на малых TF.

#### 12. Limit orders вместо market

**Текущее**: Market orders (комиссия 0.055%).

**Улучшение**: Limit orders на ask/bid (комиссия 0.02% maker).

**Экономия**: 0.035% × 2 (вход+выход) = **0.07%** на round-trip.

При 100 сделках × 27.5 USDT = экономия **~2 USDT**.
